# User Authentication System

**Date:** 2026-03-24
**Status:** Draft

## Goal

Add user authentication (Google OAuth + local credentials) alongside the existing API key auth. First user triggers an admin setup flow. User display names flow through to Slack mirroring.

## Scope

- **In scope**: Server auth endpoints, React UI (`ui/`) login/setup screens, Slack display name wiring
- **Out of scope**: Old Jinja admin dashboard (stays unprotected), `android-client/`, CLI client

## Current State

- Single static `API_KEY` env var, checked via `verify_auth()` in `app/dependencies.py`
- No user/account tables in DB
- React UI (`ui/`) stores `serverUrl` + `apiKey` in localStorage/AsyncStorage via Zustand
- UI has `AuthGate` in `app/_layout.tsx` that redirects to `/(auth)/login` when `isConfigured=false`
- Current login screen (`app/(auth)/login.tsx`) is just server URL + API key form
- Slack mirroring shows `[web]` prefix, no user attribution

## Design Decisions

### Dual auth: API key stays, user auth added alongside

```
Integration/Slack → Bearer <API_KEY>        → verify_auth() (unchanged)
React UI          → Bearer <JWT>            → verify_user() (new)
Endpoints         → verify_auth_or_user()   → accepts either
Old admin dash    → no auth (unchanged)
```

- API key = service-level access (integrations, scripts). No user identity.
- JWT = user-level access (React UI). Carries user identity.
- Chat + API v1 endpoints accept either via `verify_auth_or_user()`.
- Old `/admin/*` Jinja routes stay unprotected — not worth the effort.

### Auth methods: Google OAuth + Local

- **Google OAuth**: Server exchanges authorization code for tokens, extracts email + name + avatar from Google's userinfo endpoint. Web uses redirect flow; mobile (Expo) uses `expo-auth-session` which gives us the same auth code.
- **Local**: Email + password. Passwords hashed with bcrypt. No email verification (self-hosted, admin controls access).

### First-user setup

- If `users` table is empty, server is in "setup mode".
- `GET /auth/status` returns `{ setup_required: true, google_enabled: bool }`.
- UI shows setup screen: pick Google or local, enter display name.
- First user auto-gets `is_admin=True`.
- After first user exists, new users can only be created by admin (or invited).

### JWT structure

```json
{
  "sub": "<user_id uuid>",
  "email": "user@example.com",
  "display_name": "Alice",
  "is_admin": true,
  "exp": 1711300000
}
```

- Access token: 1 hour expiry
- Refresh token: 30 days, stored hashed in DB for revocation

## Phase 1: Database + Models

### Migration: `xxx_user_auth.py`

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR NOT NULL UNIQUE,
    display_name VARCHAR NOT NULL,
    avatar_url VARCHAR,                              -- URL to profile picture
    integration_config JSONB NOT NULL DEFAULT '{}',  -- same pattern as Bot: {"slack": {"icon_emoji": ":wave:"}}
    auth_method VARCHAR NOT NULL DEFAULT 'local',    -- 'local' | 'google'
    password_hash VARCHAR,                           -- NULL for Google-only users
    is_admin BOOLEAN NOT NULL DEFAULT FALSE,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE refresh_tokens (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash VARCHAR NOT NULL UNIQUE,
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Phase 3: Auth Router

### New file: `app/routers/auth.py`

```
GET  /auth/status        → { setup_required, google_enabled }  (no auth)
POST /auth/setup         → first-user creation                 (no auth, 409 if user exists)
POST /auth/login         → local login → { access_token, refresh_token, user }
POST /auth/google        → Google code exchange → { access_token, refresh_token, user }
POST /auth/refresh       → refresh token → { access_token }
POST /auth/logout        → revoke refresh token
GET  /auth/me            → current user profile
PUT  /auth/me            → update display_name, avatar_url
```

**Setup** (`POST /auth/setup`):
```json
// Google
{ "method": "google", "code": "4/0Abc...", "display_name": "Alice" }

// Local
{ "method": "local", "email": "admin@home.lab", "display_name": "Alice", "password": "..." }
```

**Login** (`POST /auth/login`):
```json
{ "email": "admin@home.lab", "password": "..." }
// → { "access_token": "eyJ...", "refresh_token": "...", "user": { "id": "...", "display_name": "Alice", ... } }
```

**Google** (`POST /auth/google`):
```json
{ "code": "4/0Abc...", "redirect_uri": "http://localhost:8081/auth/callback" }
// Server exchanges code → gets user info → creates/finds user → returns tokens
```

## Phase 4: Wire into Chat + Mirror

### Chat endpoints

Replace `Depends(verify_auth)` with `Depends(verify_auth_or_user)` on `/chat`, `/chat/stream`, `/chat/bots`, `/sessions/*`.

When caller is a `User` (JWT auth), inject into request handling:
- `msg_metadata["sender_name"]` = `user.display_name`
- `msg_metadata["sender_id"]` = `user:<user_id>`
- `msg_metadata["sender_avatar"]` = `user.avatar_url`

### Slack mirroring display name + icon

User model mirrors the Bot pattern: `display_name`, `avatar_url`, `integration_config` (JSONB with `{"slack": {"icon_emoji": ":wave:"}}`).

Add `user_attribution()` in `integrations/slack/client.py`, parallel to `bot_attribution()`:

```python
def user_attribution(user) -> dict:
    """Return Slack payload fields for user identity (username, icon_emoji, icon_url).
    Same pattern as bot_attribution() — uses chat:write.customize scope.
    """
    attrs: dict = {}
    if user.display_name:
        attrs["username"] = user.display_name
    slack_cfg = (user.integration_config or {}).get("slack", {})
    if slack_cfg.get("icon_emoji"):
        attrs["icon_emoji"] = slack_cfg["icon_emoji"]
    elif user.avatar_url:
        attrs["icon_url"] = user.avatar_url
    return attrs
```

Update `_mirror_to_integration()` to accept user object:

```python
async def _mirror_to_integration(
    channel, text: str, *,
    bot_id: str | None = None,
    is_user_message: bool = False,
    user = None,                    # User object when JWT-authenticated
    client_actions: list[dict] | None = None,
) -> None:
    if not channel.integration or not channel.dispatch_config:
        return
    from app.agent import dispatchers
    try:
        # For user messages with a known user: use their display name + icon
        # For bot responses: use bot attribution (existing behavior)
        # For anonymous user messages: fall back to [web] prefix
        user_attrs = {}
        if is_user_message and user:
            from integrations.slack.client import user_attribution
            user_attrs = user_attribution(user)
        elif is_user_message:
            text = f"[web] {text}"

        await dispatchers.get(channel.integration).post_message(
            channel.dispatch_config, text,
            bot_id=bot_id if not is_user_message else None,
            client_actions=client_actions,
            reply_in_thread=False,
            **user_attrs,
        )
    except Exception:
        logger.warning("Mirror to %s failed", channel.integration, exc_info=True)
```

When user is authenticated (JWT): Slack shows their `display_name` as username, their `integration_config.slack.icon_emoji` or `avatar_url` as icon. Identical to how bots get attribution.

When anonymous (API key): falls back to `[web] message` prefix as today.

### Dispatcher protocol update

Add `username`, `icon_emoji`, `icon_url` kwargs to `Dispatcher.post_message()`:

```python
class Dispatcher(Protocol):
    async def post_message(self, dispatch_config: dict, text: str, *,
                           bot_id: str | None = None,
                           username: str | None = None,
                           icon_emoji: str | None = None,
                           icon_url: str | None = None,
                           reply_in_thread: bool = True,
                           client_actions: list[dict] | None = None) -> bool: ...
```

`SlackDispatcher.post_message()`: if `username` is provided directly (from `**user_attrs`), use it instead of calling `bot_attribution()`. The existing `post_message()` in `client.py` already accepts all three kwargs — no changes needed there.

### API client (`ui/src/api/client.ts`)

- Send `Authorization: Bearer <accessToken>` (preferred) or `Bearer <apiKey>` (fallback)
- On 401: attempt refresh via `POST /auth/refresh`

### Google OAuth flow (web)

1. User clicks "Sign in with Google"
2. Redirect to Google consent screen with `GOOGLE_CLIENT_ID` (from `/auth/status`)
3. Google redirects back to `/(auth)/callback` with `?code=...`
4. UI sends code to `POST /auth/google`
5. Server exchanges code, returns tokens + user

### Google OAuth flow (Expo/mobile)

1. Use `expo-auth-session` / `expo-web-browser` to open Google consent
2. Receive auth code via redirect
3. Same `POST /auth/google` call

- **Self-edit**: `/auth/me` endpoint lets any user update their own display_name, avatar_url, integration_config (no admin required)

## Security Considerations

- Passwords hashed with bcrypt (cost factor 12)
- JWT signed with HS256 + server-side secret
- Refresh tokens stored as SHA-256 hash in DB (not plaintext)
- Google OAuth exchanges auth code server-side (client secret never exposed)
- `is_admin` checked against DB on sensitive operations, not just JWT claim
- Simple in-memory rate limit on `/auth/login` (5 attempts per email per minute)
