▶ How knowledge scoping works (bot, client, session, integrations)

Each knowledge row is matched using name (for @tags), plus bot, client, and optionally chat session for automatic semantic retrieval (RAG). Empty bot or client means “any” for that axis.

Client ID vs session ID

  • Client ID — Stable string for where the conversation lives (e.g. web client id, or integration channel id like slack:C01234567). It does not change when you open a new browser tab; for Slack it is one id per channel.
  • Session ID — UUID of a row in sessions. That row ties messages, compaction, and (for integrations) is often created to mirror the channel. Automatic knowledge search can require the current chat session to match the row’s session, or include rows with no session (legacy pool) so they appear in every thread for that bot+client.

Integrations (Slack, Discord, Teams, …)

Integration client_id values use a prefix (slack:, discord:, teams:, …). The server derives a stable session UUID from the client id alone (same channel → same UUID every time), so channel-scoped knowledge can pin to that session. Use the full editor on a row to attach or change session; the list shows the session title when the session row has one.

@name vs semantic RAG

  • Explicit @document_name — Resolves by name for the same bot + client; session is not used for that lookup.
  • Automatic retrieval — Uses bot + client + session rules above; legacy rows (session unset) still participate for that bot+client in every chat.
Filters on this page — The bot, client, and session dropdowns only narrow the table. They do not edit rows. Use Full editor to change scope or similarity per document.
