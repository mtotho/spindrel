# Track - Projects

Evergreen scope for first-class Project roots inside the singleton SharedWorkspace. Projects are named workspace-relative roots that multiple channels can attach to; they are not replacement workspaces and are not limited to one Git repo.

## Invariants

- `projects` owns the reusable root, prompt settings, and metadata; `channels.project_id` is the primary binding.
- `projects.WorkSurface` is the shared policy object for root path, display path, index root, workspace search prefix, knowledge prefix, Project prompt, and channel/project identity.
- Project-bound turns use the Project root for files, search, terminal, exec, harness cwd, context injection, and channel workspace search.
- Project-bound failures fail visibly. A broken Project binding or wrong-Project fresh instance must not silently fall back to the channel or bot workspace.
- Project Instances are disposable Project roots created from an applied Blueprint snapshot. They reuse the parent Project work-surface policy and are selected by session binding or run-scoped task context, not by mutating the channel Project binding.
- Bot-private workspace-files memory remains separate and is written through the dedicated `memory` tool.
- Project-scoped knowledge lives under `.spindrel/knowledge-base` inside the Project root.
- Project runtime env is derived from the applied Blueprint snapshot plus Project secret bindings. UI/API surfaces expose key names and readiness only; values are injected process-local into Project terminals, exec, and harness turns and are redacted from tool/harness event output.
- Project coding runs expose a single secret-safe `readiness` manifest from `ProjectTaskExecutionContext`. Agents and UI/API callers should read that manifest for runtime blockers, dependency-stack status, dev targets, handoff branch data, machine-target summary, and receipt-evidence requirements instead of reconstructing those facts from scattered Project settings.
- Provider auth has two separate lanes. Normal in-Spindrel bot turns use the Spindrel server's configured model provider or subscription provider; local development may require a one-time browser/device-code bootstrap for that local e2e server, while deployed/persistent Spindrel instances should reuse their preconfigured provider state. Future agents should check `python scripts/agent_e2e_dev.py doctor` first and skip browser OAuth when subscription bootstrap is already connected. Local e2e Postgres uses a named Docker volume and must not be wiped unless explicitly requested via `python scripts/agent_e2e_dev.py wipe-db --yes`. Codex/Claude harness turns use the harness account/config mounted or available at the Spindrel host/container's `~/.codex` / `~/.claude` paths; local e2e uses a bind-mount override to pass through the developer's host auth into the e2e container.
- **Project task execution context is owned by `app/services/project_task_execution_context.py`.** Single Module owning execution-context assembly + persistence + read accessors for Project tasks. Public Interface: `ProjectTaskExecutionContext.fresh / from_parent / review / from_task` plus `apply_to_task(task, channel)`. Adapter Protocols (`DevTargetAllocator`, `DependencyStackSpecResolver`, `ContextSource`) wire in slot-in support for follow-on Phase 4 work — per-target dependency stacks, ephemeral preview environments, non-coding-run task classes — without reshape. Continuation reuses parent context verbatim and raises `MalformedExecutionContextError` on bad parent shape; never silently re-allocates dev_targets. Round-trip `to_persisted ↔ from_task` is byte-stable. Architecture guard at `tests/unit/test_project_task_execution_context_architecture.py`. See `docs/tracks/code-quality.md` Maintenance pass 2026-05-01 for the deepening rationale.

## Project Factory Vision

This track is now the end-to-end Project Run Automation / Project Factory track, not just a Project-root settings track. The near-term product target is a general-purpose software factory where a human can capture rough notes, refine them conversationally with a normal Project-bound Codex/Claude harness channel, launch one or many Project coding runs, receive PRs and evidence, and then run review/merge sessions that mark work reviewed with provenance.

The expected conversational path is explicit: the user may enter a Codex harness channel attached to a Project, go back and forth in plan mode or normal mode to design a large multi-part track, and then ask that same Codex agent to publish the issue intakes / work packs / launchable run definitions. There is no separate "intake bot" requirement. Work-pack creation is a capability of ordinary conversational agents when they have the right Project tools and context; scheduled or operator triage is a later automation layer over the same primitives.

Locked next-phase ladder:

1. **Local dev loop correctness** — repo-dev agents lease their own native Spindrel API/UI ports via `write-env --port auto`, source `.env.agent-e2e`, use shared Docker dependencies only for Postgres/SearXNG, and never rely on a fixed local API port.
2. **Dependency stack discipline** — shared local repo-dev dependencies are treated as infrastructure, while Project coding runs use run-scoped Project Dependency Stack instances so parallel work does not restart another run's database/services.
3. **Generic Project run contract** — Project agents use native shell for the Project's own tests/scripts and app/dev servers, injected runtime/dependency env, Project Dependency Stack tools for Docker-backed services, screenshot/evidence tools, and receipts; no Spindrel-specific test runner substitutes for the Project's commands.
4. **Project runtime readiness** — every run starts with clear repo/branch/root, env/secret readiness, dependency endpoints, assigned dev-target ports, available grants/tools, screenshot target guidance, and PR/handoff intent.
5. **Generic live-loop proof** — run a generated non-Spindrel fixture Project end to end: dependency stack preflight, real harness turn, source edit, native app server on assigned port, repo-local tests, screenshot evidence, receipt, and Project Runs UI review surface.
6. **Screenshot evidence contract** — screenshots are both an agent feedback loop and proof. Development screenshots may stay as run artifacts; durable documentation proof moves into `docs/images` and is referenced by the relevant guide. Project run evidence is attached to receipts/review context even when not committed to source.
7. **Review agent loop** — review sessions can select completed runs, inspect receipts/screenshots/PRs/tests, merge accepted PRs, request follow-up, mark runs reviewed, and attach review-session provenance.
8. **Conversational issue intake to work packs** — rough notes, bugs, ideas, and planning threads can be published from normal Project-bound conversations, grouped into work packs, marked as needs-info/planning/coding, and launched as Project coding runs with source provenance.
9. **Parallel overnight factory** — multiple work packs can run overnight in fresh Project instances with isolated dependency stacks/dev targets, producing PRs, tests, screenshots, receipts, and morning review readiness.
10. **Blueprint/fresh workspace maturity** — Blueprints define repo clones, setup commands, env slots, secret bindings, dependency stacks, dev targets, and eventually per-run isolation policy. Future sidecar/container execution should extend these primitives rather than bypass them.
