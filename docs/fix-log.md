# Fix Log

- **2026-05-02** R1 Phase 2 follow-up — AST lint guard for inbound source tagging. The R1 runtime wrap is correct for today's integrations but invisible at the source-tagging layer: a typo (`"slak"`) or a new integration that forgets `msg_metadata["source"]` silently bypasses the LLM-bound `<untrusted-data>` envelope. New `tests/unit/test_integration_inbound_wrap_lint.py` AST-walks every `.py` under `integrations/` and flags any `submit_chat(...)` / `inject_message(...)` call whose source can't be resolved to a member of `EXTERNAL_UNTRUSTED_SOURCES`. Three current call shapes all resolve cleanly: `inject_message(source="<x>")` top-level kwarg (GitHub / Frigate / BlueBubbles), `submit_chat(msg_metadata={"source": "<x>"})` inline-Dict literal (Wyoming pipelines), and `submit_chat(msg_metadata=<var>)` with within-function variable trace through `Assign(Name) = Dict(...)` and `Subscript("source") = Constant` patterns (Slack / Discord). Three failure kinds (`unknown_source` / `missing_source` / `source_not_resolvable`) each become a CI failure with file:line and the resolved source value. Module-level `ALLOWED_VIOLATIONS: dict[(relpath, lineno), str]` empty at landing — every current call site resolves. Pure AST walk, no execution. Coverage: 12 cases — production-tree assertion (zero violations) + 10 synthetic-AST self-tests pinning each walker branch + allowlist short-circuit. Smoke: temporarily mutating `integrations/slack/message_handlers.py:292` to `"source": "slak"` raised the lint with `unknown_source` and the file:line.

- **2026-05-02** R1 Phase 2 follow-up — conversation-history replay wrap. R1 Phase 1 only covered the *current-turn* user message at the LLM boundary, leaving a gap: any stored Message body that came from an external source re-entered LLM context as conversation history on turn N+1 unwrapped. The wrap now fires at the canonical LLM-bound chokepoint `app/services/sessions.py::_strip_metadata_keys` — a stored user message whose `_metadata.source` is in `EXTERNAL_UNTRUSTED_SOURCES` gets the `<untrusted-data>` envelope right before the LLM sees it, regardless of whether the stored form is raw or pre-wrapped. New `app/security/prompt_sanitize.py::is_already_wrapped` checks the canonical `<untrusted-data source="` opening; `wrap_untrusted_content` now short-circuits when the input already carries the wrap (idempotent). Multimodal user turns: each text part is wrapped, image / non-text parts left intact; assistant / tool / system messages pass through untouched. Coverage: new `tests/unit/test_history_replay_wrap.py` (16 cases — idempotency contract, role gating, source gating, multimodal handling, case-insensitive source match).
