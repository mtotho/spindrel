# Integration Depth Playbook

How to bring any integration to "reference-grade citizen" on its host platform. Slack is the worked example (see [[Track - Slack Depth]]); repeat the same loop for Discord, BlueBubbles, GitHub, Web, etc.

## What "depth" means here

An integration has depth when it uses the host platform's native affordances beyond sending messages: user mentions, reactions as intents, ephemeral responses, modal forms, scheduled actions, home tabs, shortcuts, link unfurls, pins/bookmarks. Without these, we have a chat bot; with them, we have a first-class app on that platform.

The goal is not parity across every integration — it's *using everything the platform gives us*. BlueBubbles is a thin iMessage shim, so it will always have fewer affordances than Slack. That's fine; what matters is that we don't silently leave Slack affordances on the table because no one thought to wire them up.

## The loop

### 1. Audit

For the target integration, walk this checklist against the running code. Mark each: `done` / `partial` / `declared-not-implemented` / `missing`.

**Incoming affordances**
- Messages (DM + channel + mentions)
- Slash commands
- Interactive components (buttons, selects, modals)
- File uploads from user
- Reactions (user reacting as an intent signal)
- Message edits / deletions from user
- Thread replies / thread context
- App Home tab (platforms that have one)
- Shortcuts (global + message)
- Link unfurl (bot provides preview for pasted URLs)

**Outgoing affordances**
- Post text (plain + rich + long-text splitting)
- Streaming / progressive updates
- Rich layouts (Block Kit / embeds)
- File uploads from bot
- Threading
- User mentions in bot output
- Ephemeral messages
- Reactions as status indicators
- Typing / working indicator
- Self-edits
- Approval UI
- Scheduled posts
- Channel management (join/leave/invite/create)
- Proactive DMs
- Link unfurls generated by bot
- Pin / bookmark / unpin

**Meta**
- Error rendering
- Tool result rendering
- Rate limiting
- Retry / backoff
- Session/thread correlation to server channels

Record the audit in a `Track - <Integration> Depth.md` note; keep it updated as items ship.

### 2. Declare capabilities honestly

Open `integrations/<id>/integration.yaml:capabilities` and make it match what the renderer actually does. Over-declaration is a silent bug: the dispatcher routes capability-gated events to the renderer expecting it to handle them, and if the renderer has no branch, the event is silently skipped and appears to just "not work." Under-declaration is a missed opportunity.

After declaring, assert it in a renderer unit test: `Capability.X in IntegrationRenderer.capabilities`.

### 3. Ship SDK-only wins

Anything that is a pure-platform affordance — a new tool, a new event handler in the subprocess, a new branch in `renderer.render()` — is SDK-only. These don't need domain changes.

Pattern per affordance:

- **Inbound** → subprocess-side handler (`@app.event(...)` / `@app.action(...)` / `@app.shortcut(...)` in Slack Bolt). Converts the event into either a server HTTP call (for actions requiring server state) or a `submit_chat` (for a new turn).
- **Outbound event kinds** → new branch in `renderer.render()`. No new bus kind unless the *semantics* differ (ephemeral is a new kind; "post a nicer layout" is not).
- **Outbound agent actions** → new variant on `OutboundAction` + new branch in `renderer.handle_outbound_action()`. Use this when the action is agent-initiated mid-turn.
- **Agent-visible platform tools** → new file under `integrations/<id>/tools/` with `@register({...})` schema. The loader picks them up automatically.

### 4. Adopt new core capabilities

Two capabilities were introduced alongside the Slack pilot:

- **`Capability.EPHEMERAL`** — private-to-one-user delivery. Slack implementation: `chat.postEphemeral`. To adopt: declare `EPHEMERAL` in your renderer's `capabilities` and the YAML, add a branch to `render()` for `ChannelEventKind.EPHEMERAL_MESSAGE` that extracts `payload.recipient_user_id` and calls your integration's private-send API. If your platform has no private-send primitive, leave the capability off — the publisher-side fallback in `app/services/ephemeral_dispatch.py` converts the event to a broadcast `NEW_MESSAGE` with a visibility marker ("🔒 Private reply intended for @user").
- **`Capability.MODALS`** — structured input via a form UI. Slack implementation: `views.open` + `view_submission` → `POST /api/v1/modals/{callback_id}/submit`. To adopt: declare `MODALS`, wire a view_submission handler that posts to the modal submit endpoint, and (platform-specific) provide a way for the agent tool to open a modal — Slack uses a button-click trigger_id; Discord uses interaction modals. If the platform has nothing analogous, leave the capability off — the `open_modal` tool short-circuits with `unsupported=True` and the agent should ask conversationally.

Adoption steps per integration:

1. `integration.yaml:capabilities` — add the string (`ephemeral`, `modals`).
2. Renderer ClassVar `capabilities` — add the `Capability` enum member.
3. Implement the handler (`render` branch for event kinds, `handle_outbound_action` branch for actions).
4. Write a renderer unit test asserting the capability is declared AND the handler does the right thing.

### 5. Capability-gated, never special-cased

No integration-specific branches in `app/`. If you're tempted to write `if integration_id == "slack": ...` in the dispatcher or turn loop, that's a signal to either introduce a new capability or push the branch down into the integration's `integrations/<id>/` code. The `_visibility_marker` helper in `ephemeral_dispatch.py` is the only exception — it's a per-integration text format for a fallback path, not a behavior branch, and it's in a service module so the degraded UX remains integration-appropriate.

## What each integration gets "for free"

Because of the above, the following ship across every integration without per-integration work:

- Capability gating at the dispatcher (`_REQUIRED_CAPS` in `app/domain/channel_events.py`)
- Outbox-durable delivery with retry / dead-letter
- Rate limiting (shared bucket if integrations set one up in their integration dir)
- Tool approval flow (declare `APPROVAL_BUTTONS` capability + `_handle_approval_requested`)
- Streaming edit (declare `STREAMING_EDIT` + `_handle_stream_token` + `_handle_turn_ended`)
- Ephemeral fallback (declare `TEXT` and you get the "private broadcast" fallback automatically)
- Modal fallback (leave `MODALS` undeclared and the tool surfaces a clean error)

## Anti-patterns

- **Silent over-declaration** — declaring a capability in YAML but never implementing the renderer branch. Events disappear. Catch with a renderer unit test asserting each declared capability has a rendered case.
- **Dual-path delivery** — publishing the same logical event on two paths (outbox + bus) and relying on each renderer's dedup to save you. The outbox-durable vs transient split (`is_outbox_durable`) is the contract; stick with it.
- **Agent-tool reaching into integration internals** — an agent tool that imports `integrations/slack/client.py` is a dead-end; it'll never work for Discord. Tools that target "the current channel's integration" should either be scoped as `<integration_id>_<verb>` (explicitly Slack-only) or resolve the integration at runtime via `renderer_registry.get(integration_id)`.
- **Dispatcher-side rewrites** — rewriting an event from one kind to another inside `IntegrationDispatcherTask._dispatch` causes fanout trouble (the rewrite would fire per-renderer). Decisions that depend on "what integration is bound here" belong at publish time (see `deliver_ephemeral`).

## Open invariants

- `ChannelEventKind` members map 1:1 to `ChannelEventPayload` variants (enforced in `ChannelEvent.__post_init__`).
- Every new `ChannelEventKind` needs entries in `_REQUIRED_CAPS` and `_KIND_PAYLOAD` — the registration tests catch missing ones.
- Outbox-durable kinds flow through the drainer; non-durable kinds flow through the bus. Don't mix.
- Tools under `integrations/*/tools/` run server-side; they must not import from the integration subprocess code (e.g. `slack_settings.py` which hard-reads env vars).
