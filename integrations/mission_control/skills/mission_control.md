# SKILL: Mission Control

## Overview

Mission Control is a structured file protocol for managing project tasks, status, and activity using markdown files in channel workspaces. These files are readable by both humans and the Mission Control dashboard.

## Task Board (`tasks.md`)

The `tasks.md` file is a kanban board. Columns are `## ` headers, cards are `### ` headers within columns, and metadata uses `- **key**: value` format.

### Format

```markdown
# Tasks

## Backlog

### Card title here
- **id**: mc-a1b2c3
- **assigned**: bot-name
- **priority**: high
- **created**: 2026-03-30
- **tags**: backend, api
- **due**: 2026-04-15

Free text description of the task goes here.

## In Progress

### Another task
- **id**: mc-d4e5f6
- **priority**: medium
- **started**: 2026-03-28

## Review

## Done

### Completed task
- **id**: mc-g7h8i9
- **completed**: 2026-03-25
```

### Column Conventions

Standard columns (in order): **Backlog**, **In Progress**, **Review**, **Done**. You can add custom columns as needed — just use `## Column Name`.

### Card Metadata Fields

| Field | Required | Description |
|-------|----------|-------------|
| `id` | Yes | Unique ID in `mc-XXXXXX` format (auto-generated by tools) |
| `assigned` | No | Bot or person responsible |
| `priority` | No | `low`, `medium`, `high`, or `critical` |
| `created` | Yes | Date created (YYYY-MM-DD) |
| `started` | No | Date moved to In Progress |
| `completed` | No | Date moved to Done |
| `tags` | No | Comma-separated labels |
| `due` | No | Due date (YYYY-MM-DD) |

### Tools

Use these tools for task management:

- **`create_task_card`** — Creates a new card with auto-generated ID and proper metadata. Preferred over manually editing tasks.md.
- **`move_task_card`** — Moves a card between columns by ID. Automatically adds `started`/`completed` timestamps.

### When to Use tasks.md

- Tracking work items, bugs, features, action items
- Any time you need a structured list of things to do with status tracking
- Project planning with priorities and assignments

### When NOT to Use tasks.md

- Quick one-off notes → use `notes.md`
- Decision records → use `decisions.md`
- Reference material → use `references.md`

## Project Status (`status.md`)

The `status.md` file provides a quick project health summary.

### Format

```markdown
# Project Status

- **phase**: active
- **health**: green
- **updated**: 2026-03-30
- **owner**: project-bot

## Current Focus
What the project is working on right now.

## Blockers
- Waiting on API key from vendor

## Recent Milestones
- 2026-03-28: CI pipeline operational
- 2026-03-25: Project kickoff complete
```

### Health Values

- **green** — On track, no blockers
- **yellow** — Minor issues or risks
- **red** — Blocked or significantly behind

## Timeline (`timeline.md`)

The `timeline.md` file is a reverse-chronological activity log. Events are auto-logged by `move_task_card` and status changes. Use `append_timeline_event` to manually log notable events (deployments, decisions, meetings, milestones, incidents).

Format: `- HH:MM — Event description` grouped under `## YYYY-MM-DD` date headers. New entries go at the top of the current day's section.

## Best Practices

1. **Always use tools** for task creation/movement — they handle ID generation and timestamps
2. **Keep Done column clean** — archive completed tasks periodically (move card text to `archive/tasks-done.md`)
3. **Update status.md** when project health changes or milestones are reached
4. **Use consistent tags** across tasks for filtering in the dashboard
