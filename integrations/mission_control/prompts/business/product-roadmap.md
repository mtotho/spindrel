## Workspace File Organization — Product Roadmap

Organize channel workspace files for product management and roadmap planning:

- **roadmap.md** — Quarter-by-quarter feature plan
  - Now/Next/Later framework for high-level prioritization
  - Format: `## Q1 2026` with feature items, owners, target dates
  - Status per item: planned, in-progress, shipped, cut
  - Dependencies between features and cross-team coordination

- **features.md** — Feature specifications
  - Format: `### Feature Name` with structured sections
  - Problem statement, proposed solution, success metrics
  - Priority: must-have, should-have, nice-to-have
  - Size estimate: S, M, L, XL
  - Link to related roadmap quarter and user research

- **user-research.md** — User research and insights
  - Personas: key user segments with goals and pain points
  - Interview summaries: `## YYYY-MM-DD — Participant` format
  - Pain points inventory with frequency and severity
  - Feature requests ranked by demand

- **competitors.md** — Competitive landscape
  - Feature comparison matrix: `| Feature | Us | Competitor A | Competitor B |`
  - Strengths, weaknesses, and market positioning per competitor
  - Pricing and packaging analysis
  - Competitive moats and threats

- **metrics.md** — KPIs, OKRs, and key metrics
  - OKR format: `### Objective` with key results and current progress
  - Product KPI dashboard: adoption, engagement, retention, revenue
  - Feature-specific success metrics and targets
  - Update cadence: weekly for KPIs, quarterly for OKRs

- **tasks.md** — Kanban board for product work items (Mission Control compatible)

- **status.md** — Product health and priorities (Mission Control compatible)

- **timeline.md** — Reverse-chronological activity log (Mission Control compatible)
  - Auto-captures product events: feature launches, roadmap changes, research sessions, metric milestones
  - Entries: `- HH:MM — Event description` grouped under `## YYYY-MM-DD` date headers
  - Append new entries at top of current day's section
  - Use `append_timeline_event` tool to log events (also auto-logged by `move_task_card`, status changes, etc.)

- **notes.md** — Meeting notes, brainstorm captures, scratch work
  - Date-stamp entries: `## YYYY-MM-DD Topic`
  - Product review outcomes, stakeholder feedback
  - Action items in bold or checklist format

Create files as needed — not all files are required from the start. Archive shipped feature specs and old research to the archive/ folder.

### tasks.md — Kanban Format

Use a markdown kanban board for product management tasks:

```markdown
## Ideas

### In-app onboarding wizard for new users
- **id**: mc-a1b2c3
- **priority**: high
- **tags**: onboarding, growth

### Self-service team management portal
- **id**: mc-s9t0u1
- **priority**: medium
- **tags**: enterprise, self-serve

## Scoping

### Advanced analytics dashboard
- **id**: mc-d4e5f6
- **priority**: high
- **tags**: analytics, enterprise
- **started**: 2026-03-28

## Building

### Collaborative editing for documents
- **id**: mc-g7h8i9
- **priority**: critical
- **tags**: core, collaboration
- **started**: 2026-03-20

## Shipped

### API rate limiting and usage dashboard
- **id**: mc-j0k1l2
- **priority**: high
- **tags**: api, platform
- **completed**: 2026-03-25
```

Each card follows the Mission Control card format:
- **id** (required): Unique `mc-XXXXXX` identifier (auto-generated by `create_task_card`)
- **priority**: `low`, `medium`, `high`, or `critical`
- **tags**: Comma-separated labels (e.g. onboarding, enterprise, core)
- **started** / **completed**: ISO date timestamps (auto-set by `move_task_card`)

Use `create_task_card` and `move_task_card` tools for all task management — tasks.md is a read-only rendering from the database and must not be edited directly.

### status.md — Product Status Format

```markdown
- **phase**: Q2 2026 — Growth & Enterprise
- **health**: green
- **updated**: 2026-03-28
- **owner**: Product Lead

## Current Focus
- Collaborative editing shipping end of sprint (critical feature)
- Analytics dashboard in scoping — stakeholder interviews this week

## Blockers
- None currently

## Recent Milestones
- 2026-03-25: API rate limiting shipped — enterprise pilot started
- 2026-03-18: Q2 roadmap finalized — 12 features across 3 themes
```

Health values: `green` (roadmap on track, key features progressing), `yellow` (features behind schedule or scope creep), `red` (major features blocked or quarter at risk).

### timeline.md — Activity Log

Reverse-chronological event stream capturing product management activity. New entries go at the top of the current day's section.

```markdown
## 2026-03-28

- 16:30 — Card mc-d4e5f6 moved to **Scoping** (was: Ideas) — "Advanced analytics dashboard"
- 14:00 — User research: 3 enterprise interviews completed — top request: granular permissions
- 11:45 — Competitor update: Competitor A launched AI-powered search feature
- 09:30 — Q2 OKR check-in: 2/4 key results on track, 2 at risk (adoption metrics)

## 2026-03-27

- 17:00 — Card mc-j0k1l2 moved to **Shipped** (was: Building) — "API rate limiting and usage dashboard"
- 15:00 — Feature spec finalized: "Advanced analytics dashboard" — 3 stakeholder sign-offs
- 10:00 — Metrics update: weekly active users +8% MoM, retention steady at 72%
```

Events are auto-logged by `move_task_card` and status changes. Use `append_timeline_event` to manually log notable events (feature launches, roadmap pivots, research findings, metric milestones, competitive intel). Heartbeats can auto-append periodic entries for passive monitoring.

### plans.md — Structured Execution Plans (Read-Only Rendering)

Plans are stored in the MC database. `plans.md` is auto-generated after every state change — never edit it directly. Use `draft_plan` to create plans, and `update_plan_step`/`update_plan_status` for mutations. After approval, the plan executor automatically sequences step execution. Pull the planning skill for the full protocol.
