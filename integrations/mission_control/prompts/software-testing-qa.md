## Workspace File Organization — Software Testing / QA

Organize channel workspace files to track testing activities:

- **test-plan.md** — Test strategy and scope
  - Features/modules under test, out-of-scope items
  - Test types: unit, integration, E2E, regression, performance
  - Environment requirements, test data needs
  - Entry/exit criteria, risk areas

- **test-cases.md** — Test case definitions
  - Format: `### TC-NNN: Title` with steps, expected results, priority
  - Group by feature or module using `## Feature Name` headers
  - Mark status: draft, ready, automated, deprecated

- **tasks.md** — Kanban board for QA tasks (Mission Control compatible)

- **status.md** — Testing progress and health (Mission Control compatible)

- **bugs.md** — Bug tracker
  - Format: `### BUG-NNN: Title` with severity, steps to reproduce, expected vs actual
  - Status: open, investigating, fixed, verified, closed
  - Link to related test cases (TC-NNN references)

- **coverage.md** — Test coverage analysis
  - Coverage matrix: feature vs test type
  - Gaps and untested areas
  - Automation coverage percentage and targets

- **results.md** — Test execution results
  - Append new runs with date headers: `## YYYY-MM-DD Run`
  - Summary: total/pass/fail/skip counts
  - Failed test details and investigation notes

- **notes.md** — Working notes and scratch space
  - Environment quirks, workarounds, tribal knowledge
  - Meeting notes from test planning/triage sessions

Create files as needed — not all files are required from the start. Archive old test results and closed bugs to the archive/ folder.

### tasks.md — Kanban Format

Use a markdown kanban board for QA tasks:

```markdown
## Backlog

### Write E2E tests for checkout flow
- **id**: mc-a1b2c3
- **priority**: high
- **tags**: e2e, checkout

## In Progress

### Investigate flaky login test
- **id**: mc-d4e5f6
- **priority**: critical
- **tags**: bug, flaky-test
- **started**: 2026-03-28

## Review

### Review regression suite for v2.1
- **id**: mc-g7h8i9
- **priority**: high
- **tags**: regression, release
- **started**: 2026-03-25

## Done

### Set up test environment for staging
- **id**: mc-j0k1l2
- **priority**: high
- **completed**: 2026-03-20
```

Each card follows the Mission Control card format:
- **id** (required): Unique `mc-XXXXXX` identifier (auto-generated by `create_task_card`)
- **priority**: `low`, `medium`, `high`, or `critical`
- **tags**: Comma-separated labels (e.g. e2e, regression, bug, flaky-test)
- **started** / **completed**: ISO date timestamps (auto-set by `move_task_card`)

Use `create_task_card` and `move_task_card` tools for task management instead of manual edits.

### status.md — QA Status Format

```markdown
- **phase**: Release Testing — v2.1
- **health**: yellow
- **updated**: 2026-03-28
- **owner**: QA Lead

## Current Focus
- Regression testing for v2.1 release
- Investigating 2 flaky tests in CI

## Blockers
- Staging environment intermittently unavailable

## Recent Milestones
- 2026-03-20: Test environment provisioned
- 2026-03-15: Test plan approved for v2.1
```

Health values: `green` (all tests passing, on track), `yellow` (some failures or blockers), `red` (critical bugs or blocked).
