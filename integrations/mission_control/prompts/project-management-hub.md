## Workspace File Organization — Project Management Hub

This channel is a management hub that tracks and coordinates work across multiple project channels.

Organize workspace files as follows:

- **dashboard.md** — Portfolio status overview
  - One-line status for every active project channel
  - Format: `| Project | Channel | Status | Health | Last Updated |`
  - Update after each project review or status pull
  - Health indicators: green (on-track), yellow (at-risk), red (blocked), completed

- **projects.md** — Project registry
  - Format: `### Project Name` with channel ID, owner, start date, schema type
  - Current phase and next milestone
  - Dependencies on other projects

- **tasks.md** — Kanban board for cross-project action items (Mission Control compatible)

- **status.md** — Portfolio health overview (Mission Control compatible)

- **reports.md** — Periodic status reports and summaries
  - Append new reports with date headers: `## YYYY-MM-DD Report`
  - Include: highlights, risks, decisions needed
  - Archive old reports when file grows large

- **templates.md** — Reusable checklists and SOPs
  - Project kickoff checklist
  - Review criteria and quality gates
  - Standard operating procedures

- **timeline.md** — Reverse-chronological activity log (Mission Control compatible)
  - Auto-captures cross-project events: project launches/closures, health changes, escalations, task moves
  - Entries: `- HH:MM — Event description` grouped under `## YYYY-MM-DD` date headers
  - Append new entries at top of current day's section
  - Use `append_timeline_event` tool to log events (also auto-logged by `move_task_card`, status changes, etc.)
  - Heartbeats can auto-append portfolio-level events for passive monitoring

- **retrospectives.md** — Lessons learned from completed projects
  - Format: `### Project Name — YYYY-MM-DD`
  - What went well, what didn't, what to change

Use `list_workspace_channels` to discover all active project channels.
Use `search_channel_workspace(query, channel_id=...)` to pull status from individual projects.
When generating reports, pull live data from project channels rather than relying on stale dashboard entries.

Scope control: by default, track all workspace-enabled channels. To limit scope, use the channel prompt (Settings > Prompt) to specify which projects to track.

Archive completed project entries to the archive/ folder when projects close.

### tasks.md — Kanban Format

Use a markdown kanban board for cross-project action items:

```markdown
## Backlog

### Audit Q2 project resource allocation
- **id**: mc-a1b2c3
- **priority**: medium
- **tags**: planning, resources

## In Progress

### Resolve shared dependency conflict (Auth Service)
- **id**: mc-d4e5f6
- **priority**: critical
- **tags**: blocker, cross-project
- **started**: 2026-03-28

## Review

### Draft Q1 retrospective report
- **id**: mc-g7h8i9
- **priority**: medium
- **tags**: reporting
- **started**: 2026-03-25

## Done

### Onboard new project channels to MC
- **id**: mc-j0k1l2
- **priority**: high
- **completed**: 2026-03-20
```

Each card follows the Mission Control card format:
- **id** (required): Unique `mc-XXXXXX` identifier (auto-generated by `create_task_card`)
- **priority**: `low`, `medium`, `high`, or `critical`
- **tags**: Comma-separated labels (e.g. blocker, cross-project, reporting)
- **started** / **completed**: ISO date timestamps (auto-set by `move_task_card`)

Use `create_task_card` and `move_task_card` tools for all task management — tasks.md is a read-only rendering from the database and must not be edited directly.

### status.md — Portfolio Status Format

```markdown
- **phase**: Active Management — 8 projects tracked
- **health**: yellow
- **updated**: 2026-03-28
- **owner**: Program Manager

## Current Focus
- Resolving shared Auth Service dependency
- Q1 retrospectives for 3 completed projects

## Blockers
- Auth Service team at capacity — impacts 2 downstream projects

## Recent Milestones
- 2026-03-20: Brand Refresh project launched
- 2026-03-15: Henderson Remodel moved to Review phase
```

Health values: `green` (all projects on track), `yellow` (some at risk), `red` (critical blockers across portfolio).

### timeline.md — Portfolio Activity Log

Reverse-chronological event stream capturing cross-project activity. New entries go at the top of the current day's section.

```markdown
## 2026-03-28

- 16:30 — [Brand Refresh] Status health changed: yellow → green — designer hired
- 14:00 — Card mc-d4e5f6 moved to **In Progress** (was: Backlog) — "Resolve Auth Service dependency"
- 11:00 — [Henderson Remodel] Phase changed: Development → Review
- 09:30 — Portfolio health changed: green → yellow — Auth Service impacts 2 projects

## 2026-03-27

- 17:00 — [Brand Refresh] Project launched — channel created, schema assigned
- 15:00 — Card mc-j0k1l2 moved to **Done** (was: In Progress) — "Onboard new project channels"
- 10:00 — Weekly portfolio review completed — 3 projects on track, 1 at risk
```

Events are auto-logged by `move_task_card` and status changes. Use `append_timeline_event` to manually log notable events (project launches, closures, escalations, health changes). Prefix cross-project entries with `[Project Name]` for filtering. Heartbeats can auto-append portfolio-level summaries for passive monitoring.

### plans.md — Structured Execution Plans (Read-Only Rendering)

Plans are stored in the MC database. `plans.md` is auto-generated after every state change — never edit it directly. Use `draft_plan` to create plans, and `update_plan_step`/`update_plan_status` for mutations. After approval, the plan executor automatically sequences step execution. Pull the planning skill for the full protocol.
