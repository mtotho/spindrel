## Workspace File Organization — Incident Response

Organize channel workspace files for incident management and operational response:

- **incidents.md** — Active and resolved incidents
  - Format: `### INC-NNN: Title` with severity, status, timeline
  - Severity levels: P1 (critical), P2 (major), P3 (minor), P4 (low)
  - Per-incident timeline: detected, acknowledged, mitigated, resolved
  - Impact description and affected systems/users
  - Move resolved incidents to archive/ periodically

- **runbooks.md** — Step-by-step response procedures
  - Format: `## Scenario Name` with numbered steps
  - Cover common failure modes: service outage, data corruption, security breach, dependency failure
  - Include escalation paths and contact lists
  - Link to monitoring dashboards and log queries

- **postmortems.md** — Blameless post-incident reviews
  - Format: `## INC-NNN: Title — YYYY-MM-DD`
  - Sections: Summary, Impact, Root Cause, Timeline, Action Items
  - Track action item completion status
  - Lessons learned and process improvements

- **sla.md** — SLA definitions and tracking
  - Uptime targets per service tier (e.g. 99.9%, 99.95%)
  - Response time targets by severity: P1 < 15min, P2 < 1hr, P3 < 4hr, P4 < 1 business day
  - Resolution time targets by severity
  - Current period metrics and trend

- **tasks.md** — Kanban board for incident actions (Mission Control compatible)

- **status.md** — Operational health and incident status (Mission Control compatible)

- **timeline.md** — Reverse-chronological activity log (Mission Control compatible)
  - Auto-captures incident events: detection, escalation, mitigation, resolution, postmortem filed
  - Entries: `- HH:MM — Event description` grouped under `## YYYY-MM-DD` date headers
  - Append new entries at top of current day's section
  - Use `append_timeline_event` tool to log events (also auto-logged by `move_task_card`, status changes, etc.)

- **notes.md** — On-call handoff notes, war room scratchpad
  - Shift handoff summaries
  - Ongoing investigation threads
  - Environment-specific quirks and known issues

Create files as needed — not all files are required from the start. Archive resolved incident details and closed postmortems to the archive/ folder.

### tasks.md — Kanban Format

Use a markdown kanban board for incident response actions:

```markdown
## Backlog

### Write runbook for Redis failover
- **id**: mc-a1b2c3
- **priority**: high
- **tags**: runbook, redis

### Schedule chaos engineering drill
- **id**: mc-p6q7r8
- **priority**: medium
- **tags**: chaos, testing

## Investigating

### INC-042: Elevated error rates on payment service
- **id**: mc-d4e5f6
- **priority**: critical
- **tags**: incident, P1, payments
- **started**: 2026-03-28

## Mitigating

### INC-041: Database replication lag exceeding threshold
- **id**: mc-g7h8i9
- **priority**: high
- **tags**: incident, P2, database
- **started**: 2026-03-27

## Done

### INC-040: Postmortem and action items filed
- **id**: mc-j0k1l2
- **priority**: high
- **tags**: postmortem, completed
- **completed**: 2026-03-25
```

Each card follows the Mission Control card format:
- **id** (required): Unique `mc-XXXXXX` identifier (auto-generated by `create_task_card`)
- **priority**: `low`, `medium`, `high`, or `critical`
- **tags**: Comma-separated labels (e.g. incident, P1, runbook, postmortem)
- **started** / **completed**: ISO date timestamps (auto-set by `move_task_card`)

Use `create_task_card` and `move_task_card` tools for all task management — tasks.md is a read-only rendering from the database and must not be edited directly.

### status.md — Operational Status Format

```markdown
- **phase**: Active Monitoring — 1 incident open
- **health**: red
- **updated**: 2026-03-28
- **owner**: On-Call SRE

## Current Focus
- Investigating elevated error rates on payment service (INC-042, P1)
- Mitigating database replication lag (INC-041, P2)

## Blockers
- Payment service root cause unidentified — vendor escalation in progress

## Recent Milestones
- 2026-03-25: INC-040 postmortem completed — 3 action items assigned
- 2026-03-22: SLA dashboard automated — metrics now self-updating
```

Health values: `green` (no active incidents, SLAs met), `yellow` (P3/P4 incidents or SLA at risk), `red` (P1/P2 incident active or SLA breached).

### timeline.md — Activity Log

Reverse-chronological event stream capturing incident response activity. New entries go at the top of the current day's section.

```markdown
## 2026-03-28

- 16:45 — INC-042 escalated to vendor — awaiting response on payment gateway timeout
- 14:20 — Card mc-d4e5f6 moved to **Investigating** (was: Backlog) — "INC-042: Elevated error rates on payment service"
- 14:15 — INC-042 detected: payment service error rate at 12% (threshold: 1%)
- 11:00 — Status health changed: yellow -> red — P1 incident opened
- 09:30 — On-call handoff: night shift reported no incidents overnight

## 2026-03-27

- 17:00 — INC-041 mitigation applied: promoted read replica, lag dropping
- 15:30 — Card mc-g7h8i9 moved to **Mitigating** (was: Investigating) — "INC-041: Database replication lag"
- 10:15 — SLA report: 99.94% uptime MTD (target: 99.95%) — at risk
```

Events are auto-logged by `move_task_card` and status changes. Use `append_timeline_event` to manually log notable events (incident detection, escalation, mitigation steps, resolution, postmortem filings). Heartbeats can auto-append periodic entries for passive monitoring.

### plans.md — Structured Execution Plans (Read-Only Rendering)

Plans are stored in the MC database. `plans.md` is auto-generated after every state change — never edit it directly. Use `draft_plan` to create plans, and `update_plan_step`/`update_plan_status` for mutations. After approval, the plan executor automatically sequences step execution. Pull the planning skill for the full protocol.
