## Workspace File Organization — Sprint / Agile

Organize channel workspace files to manage agile sprints:

- **sprint.md** — Current sprint definition
  - Sprint number, start/end dates, goal statement
  - Commitment: total story points committed vs capacity
  - Velocity history (last 3-5 sprints) for planning reference
  - Sprint-specific risks or dependencies

- **backlog.md** — Prioritized product backlog
  - Stories ordered by priority with story point estimates
  - Format: `### Story Title` with acceptance criteria, points, priority
  - Epics as `## Epic Name` headers grouping related stories
  - Refinement status: unrefined, ready, needs-split

- **retro.md** — Sprint retrospectives
  - Format: `## Sprint N — YYYY-MM-DD`
  - What went well, what went badly, action items
  - Track action item follow-through across sprints
  - Velocity trend notes

- **standup.md** — Daily standup log
  - Format: `## YYYY-MM-DD` with per-person updates
  - Yesterday, today, blockers format
  - Keep recent (archive after sprint ends)

- **tasks.md** — Kanban board for sprint work items (Mission Control compatible)

- **status.md** — Sprint health and progress (Mission Control compatible)

- **timeline.md** — Reverse-chronological activity log (Mission Control compatible)
  - Auto-captures sprint events: card moves, sprint starts/ends, velocity updates, blocker flags
  - Entries: `- HH:MM — Event description` grouped under `## YYYY-MM-DD` date headers
  - Append new entries at top of current day's section
  - Use `append_timeline_event` tool to log events (also auto-logged by `move_task_card`, status changes, etc.)

- **notes.md** — Meeting notes, planning sessions, scratch work
  - Date-stamp entries: `## YYYY-MM-DD Topic`
  - Sprint planning notes, grooming session outcomes
  - Action items in bold or checklist format

Create files as needed — not all files are required from the start. Archive completed sprint retros and old standup logs to the archive/ folder.

### tasks.md — Kanban Format

Use a markdown kanban board with sprint-specific columns:

```markdown
## Sprint Backlog

### Implement password reset flow
- **id**: mc-a1b2c3
- **priority**: high
- **tags**: auth, frontend
- **points**: 5

### Add rate limiting to API endpoints
- **id**: mc-m3n4o5
- **priority**: medium
- **tags**: backend, security
- **points**: 3

## In Dev

### Build notification preferences page
- **id**: mc-d4e5f6
- **priority**: high
- **tags**: frontend, notifications
- **points**: 8
- **started**: 2026-03-28

## In Review

### Refactor database connection pooling
- **id**: mc-g7h8i9
- **priority**: medium
- **tags**: backend, infrastructure
- **points**: 5
- **started**: 2026-03-25

## Done

### Set up feature flag service
- **id**: mc-j0k1l2
- **priority**: high
- **points**: 3
- **completed**: 2026-03-20
```

Each card follows the Mission Control card format:
- **id** (required): Unique `mc-XXXXXX` identifier (auto-generated by `create_task_card`)
- **priority**: `low`, `medium`, `high`, or `critical`
- **tags**: Comma-separated labels for categorization
- **points**: Story point estimate (Fibonacci: 1, 2, 3, 5, 8, 13)
- **started** / **completed**: ISO date timestamps (auto-set by `move_task_card`)

Use `create_task_card` and `move_task_card` tools for all task management — tasks.md is a read-only rendering from the database and must not be edited directly.

### status.md — Sprint Status Format

```markdown
- **phase**: Sprint 7 — API Hardening
- **health**: yellow
- **updated**: 2026-03-28
- **owner**: Scrum Master

## Current Focus
- Completing notification preferences (8 pts in dev)
- Database connection pooling in review (5 pts)

## Blockers
- Design assets for password reset delayed — blocks 5-point story

## Recent Milestones
- 2026-03-20: Sprint 7 started — 21 points committed (velocity avg: 19)
- 2026-03-18: Sprint 6 closed — 18/20 points delivered
```

Health values: `green` (on track for sprint goal), `yellow` (at risk — velocity behind or blockers), `red` (sprint goal compromised).

### timeline.md — Activity Log

Reverse-chronological event stream capturing sprint activity. New entries go at the top of the current day's section.

```markdown
## 2026-03-28

- 16:45 — Card mc-g7h8i9 moved to **In Review** (was: In Dev) — "Refactor database connection pooling"
- 14:20 — Status health changed: green -> yellow — design assets delayed
- 11:00 — Standup: 2 items in dev, 1 blocker flagged (design assets)
- 09:30 — Card mc-d4e5f6 moved to **In Dev** (was: Sprint Backlog) — "Build notification preferences page"

## 2026-03-27

- 17:00 — Sprint burndown: 13/21 points remaining (day 8 of 10)
- 15:30 — New card created: mc-m3n4o5 "Add rate limiting to API endpoints" (3 pts)
- 10:15 — Card mc-j0k1l2 moved to **Done** (was: In Review) — "Set up feature flag service"
```

Events are auto-logged by `move_task_card` and status changes. Use `append_timeline_event` to manually log notable events (sprint starts/ends, velocity snapshots, blocker updates, planning outcomes). Heartbeats can auto-append periodic entries for passive monitoring.

### plans.md — Structured Execution Plans (Read-Only Rendering)

Plans are stored in the MC database. `plans.md` is auto-generated after every state change — never edit it directly. Use `draft_plan` to create plans, and `update_plan_step`/`update_plan_status` for mutations. After approval, the plan executor automatically sequences step execution. Pull the planning skill for the full protocol.
